# Real Data Execution Plan

## Current State

The codebase is in a mixed state:

- `backend/main.py` still loads the full catalog and synthetic history on startup through `load_all_cs2_data()`.
- `backend/collectors/real_data_collector.py` does collect live Steam data, but the lifecycle, observability, and source coverage still need hardening.
- `frontend/app/market/page.tsx` is still sample-driven.
- `frontend/app/items/[id]/page.tsx` still mixes mock chart data with live multi-source fetches.
- `backend/routers/items.py` exposes a multi-source prices route, but the backend currently only has real Steam ingestion.

This plan turns that into a clear production path.

## Definition Of Done

The platform is considered to have "real data" when:

- Production startup does not generate synthetic history.
- Steam live snapshots are stored reliably and are visible in the UI.
- Historical charts are clearly labeled if they are incomplete or demo-generated.
- Frontend market and item pages read from the backend API instead of hardcoded sample data.
- Any source shown in the UI has a real ingestion path or is removed/hidden.
- Admin and verification endpoints show collection health, last success time, and data volume.

## Phase 1: Split Demo Bootstrap From Production Bootstrap

- Effort: `0.5-1 day`
- Goal: stop conflating demo seed data with production data.
- Work:
  - Make startup behavior environment-driven.
  - Keep synthetic seeding available for local/demo mode only.
  - In production mode, initialize the schema and start live collection without generating fake history.
  - Make it explicit in code and docs when history is synthetic, demo-only, or live.
- Exit criteria:
  - A fresh production deploy can start with no synthetic backfill.
  - Demo mode still works for local development.
  - No doc claims synthetic history is real history.

## Phase 2: Harden Steam Live Collection

- Effort: `1-2 days`
- Goal: make real Steam collection observable and trustworthy.
- Work:
  - Verify `backend/collectors/real_data_collector.py` writes valid `PriceHistory` rows and uses `source="steam"`.
  - Confirm `backend/collectors/steam_market.py` resolves market hash names correctly in a production-like run.
  - Make collector start/stop behavior idempotent so startup does not spawn duplicate loops.
  - Record last successful collection time, success count, failure count, and duration.
  - Surface those metrics through admin endpoints.
- Exit criteria:
  - Manual collection produces real database rows.
  - Scheduled collection can be proven from logs or admin stats.
  - Failures are visible, not silent.

## Phase 3: Fix Historical Data Semantics

- Effort: `1-3 days`
- Goal: stop representing generated history as if it were market history.
- Work:
  - Treat the history generated by `backend/collectors/comprehensive_loader.py` as demo/backfill data, not real market data.
  - Add a clear source or mode distinction so charts can tell live rows from synthetic rows.
  - Audit schema mismatches and loader assumptions before relying on historical backfill in production.
  - Fix any model/loader inconsistencies that prevent stable writes or analytics.
- Exit criteria:
  - Charts and API responses distinguish live versus synthetic data.
  - No production path silently fabricates "historical" prices.

## Phase 4: Remove Frontend Mock Dependence

- Effort: `1-3 days`
- Goal: make the UI read backend data instead of hardcoded examples.
- Work:
  - Replace `frontend/app/market/page.tsx` sample rows with `/items` and `/items/trending` API data.
  - Replace the item detail mock datasets in `frontend/app/items/[id]/page.tsx` with backend item, price history, trend, and prediction endpoints.
  - Keep loading skeletons and empty states, but remove fake prices from normal rendering paths.
  - Preserve a fallback state for failed requests that does not invent market data.
- Exit criteria:
  - The market page renders backend data only.
  - Item detail pages render backend history only.
  - No visible UI section depends on fixed sample prices.

## Phase 5: Decide Multi-Source Reality

- Effort: `2-5 days` for plumbing only, `1-2 weeks` if adding real external ingest
- Goal: align the source selector with actual data availability.
- Work:
  - Either implement real collectors for the non-Steam sources shown in the UI, or remove those options from the UX.
  - Keep the API contract honest: every source in the chart or filter should have real data behind it.
  - Normalize source tags, timestamps, and row semantics if multiple providers are added.
- Exit criteria:
  - Each UI source maps to a real ingestion path, or it is hidden/removed.
  - The backend and frontend agree on what sources exist.

## Phase 6: Add Verification And Data Quality Checks

- Effort: `1-2 days`
- Goal: make it obvious when the real-data path breaks.
- Work:
  - Add a status endpoint or dashboard metrics for last successful collection, total items, total price records, and source breakdown.
  - Add tests for the live collection path.
  - Add a test that fails if production bootstrap falls back to synthetic history.
  - Add a test for the frontend API contract so mock data cannot silently reappear.
- Exit criteria:
  - A broken collector is visible quickly.
  - A misconfigured production deploy cannot look healthy while serving fake history.

## Phase 7: Update Documentation To Match Reality

- Effort: `2-4 hours`
- Goal: make the docs tell the truth about the current implementation.
- Work:
  - Update `IMPLEMENTATION_STATUS.md` to distinguish complete backend plumbing from real-world data coverage.
  - Update `REAL_DATA_COLLECTION.md` to state which parts are live, which are synthetic, and which are demo-only.
  - Update the README and any setup docs so startup behavior is consistent with the selected mode.
- Exit criteria:
  - A new reader can tell exactly what is live today.
  - The docs do not overstate multi-source support or historical realism.

## Recommended Order

1. Phase 1
2. Phase 2
3. Phase 3
4. Phase 4
5. Phase 6
6. Phase 7
7. Phase 5 only if multi-source ingestion is a real near-term requirement

The reason for that order is simple: first make the live Steam path trustworthy, then remove synthetic and mock dependencies, then decide whether extra sources are worth the complexity.

## Practical Estimate

- To get to "real Steam data is visible in the app": `2-5 days`
- To get to "no production reliance on mock or synthetic market data": `1-2 weeks`
- To get to "multiple real external sources in production": add `1-2 weeks` depending on access, rate limits, and API stability

## Acceptance Checklist

- [ ] Production startup does not generate synthetic history
- [ ] Live Steam rows are persisted with a clear source tag
- [ ] Admin status shows last successful collection
- [ ] Market page uses backend data
- [ ] Item detail page uses backend price history
- [ ] Multi-source UI only exposes real sources
- [ ] Docs describe the current state accurately
